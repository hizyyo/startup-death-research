## 150-Word Version

The Anatomy of Startup Death is an independent research project that studies early-stage startup failure using public postmortem evidence. I built a 100-case dataset from founder postmortems, interviews, shutdown announcements, and detailed reporting. For each case, I coded startup category, customer type, business model, stage, geography, source quality, confidence level, primary failure cause, secondary causes, and evidence summary.

The project includes a custom failure taxonomy, dataset schema, methodology, source pipeline, validation checks, batch notes, summary statistics, limitations, and a research memo. The dataset currently contains 72 primary-source cases and 69 high-confidence cases with zero schema validation errors.

My goal was to move beyond anecdotal startup advice and build a transparent research foundation for studying patterns in startup failure. The first analysis suggests that many failures come from mismatches between product, urgent customer need, monetization, operations, platform dependence, and founder motivation.

## Intellectual Motivation

Startup failure is often discussed through clichés: build something people want, do things that do not scale, raise more money, move fast.

I wanted to test whether public startup failure stories could be converted into structured data. Instead of reading postmortems as isolated stories, I classified them using a consistent schema to see which patterns repeated.

The project became a bridge between entrepreneurship, data, and social science. It required both technical structure and interpretive judgment.

## What I Learned

The most important lesson is that failure is usually multi-causal.

Many startups did not fail only because the product was bad. They failed because usage did not become monetization, because the problem was not urgent, because marketplaces could not balance supply and demand, because operations were too complex, because platforms changed rules, or because founders lost alignment.

I also learned that serious research requires restraint. A dataset can reveal patterns without claiming to explain everything.

## How To Discuss In Interviews

If asked what the project is:

> I built a structured dataset of 100 startup failure cases using public postmortems and founder interviews. The goal was to turn anecdotal failure stories into analyzable research data.

If asked why it matters:

> Startup failure is usually discussed as advice or storytelling. I wanted to create a more systematic view by coding failures with a transparent taxonomy and evidence summaries.

If asked what was difficult:

> The hardest part was classification. Most failures have multiple causes, so I had to design a taxonomy, separate primary from secondary causes, and be honest about uncertainty.

If asked what the first finding was:

> Demand failure dominates, but the interesting pattern is that demand failure is not just "bad product." It often means the problem was not urgent, the business model did not work, or usage did not convert into retention and revenue.

## Avoid Overclaiming

Do not say:

> I discovered why startups fail.

Better:

> I built a structured dataset to study patterns in how startup failures are described and evidenced in public postmortems.

Do not say:

> My dataset predicts startup failure.

Better:

> My dataset is descriptive, not predictive. It helps identify recurring patterns in public failure narratives.
